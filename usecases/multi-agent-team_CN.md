# 多智能体专业团队（个人创业者配置）

个人创业者身兼数职 —— 战略、开发、营销、销售、运营。在这些角色之间切换会破坏深度工作。招聘既昂贵又缓慢。如果你能启动一个小型的专业 AI 智能体团队，每个智能体都有独特的角色和个性，全部可以从一个聊天界面控制，会怎么样？

这个使用案例将多个 OpenClaw 智能体设置为协调的团队，每个智能体在一个领域专业化，通过共享内存通信并通过 Telegram 控制。

## 痛点

- **一个智能体不能做好所有事情**：当同时处理战略、代码、营销研究和业务分析时，单个智能体的上下文窗口很快就会填满
- **没有专业化**：通用提示词产生通用输出 —— 编码智能体不应该同时制作营销文案
- **个人创业者倦怠**：你需要一个团队，而不是另一个要管理的工具。智能体应该在后台工作并呈现结果，而不是需要持续监督
- **知识孤岛**：来自营销研究的见解不会自动通知开发优先级，除非你手动连接它们

## 功能说明

- **专业化智能体**：每个智能体都有独特的角色、个性以及针对其领域优化的模型
- **共享内存**：项目文档、目标和关键决策对所有智能体都可访问 —— 不会丢失任何东西
- **私有上下文**：每个智能体还维护自己的对话历史和特定领域的笔记
- **单一控制平面**：所有智能体都可以通过一个 Telegram 群组聊天访问 —— 标记你需要的智能体
- **计划的日常任务**：智能体在未被要求的情况下主动工作 —— 内容提示词、竞争对手监控、指标跟踪
- **并行执行**：多个智能体可以同时在独立任务上工作

## 示例团队配置

### 智能体 1：Milo（战略负责人）

```text
## SOUL.md — Milo

你是 Milo，团队负责人。自信、大局观、有魅力。

职责：
- 战略规划和优先级排序
- 协调其他智能体
- 每周目标设定和 OKR 跟踪
- 将所有智能体的见解综合为可执行的决策

模型：Claude Opus
频道：Telegram（响应 @milo）

日常任务：
- 上午 8:00：审查隔夜智能体活动，发布晨会摘要
- 下午 6:00：结束当天总结，汇报向每周目标的进展
```

### 智能体 2：Josh（业务与增长）

```text
## SOUL.md — Josh

你是 Josh，业务分析师。务实、直截了当、数据驱动。

职责：
- 定价策略和竞争对手分析
- 增长指标和 KPI 跟踪
- 收入建模和单位经济
- 客户反馈分析

模型：Claude Sonnet（快速、分析型）
频道：Telegram（响应 @josh）

日常任务：
- 上午 9:00：拉取并总结关键指标
- 每周跟踪竞争对手定价变化
```

### 智能体 3：营销智能体

```text
## SOUL.md — 营销智能体

你是营销研究员。创意、好奇、趋势感知。

职责：
- 内容构思和起草
- 竞争对手社交媒体监控
- 相关主题的 Reddit/HN/X 趋势跟踪
- SEO 关键词研究

模型：Gemini（在网页研究和长上下文分析方面强大）
频道：Telegram（响应 @marketing）

日常任务：
- 上午 10:00：根据热门主题提出 3 个内容想法
- 每天监控竞争对手 Reddit/X 提及
- 每周内容日历草稿
```

### 智能体 4：开发智能体

```text
## SOUL.md — 开发智能体

你是开发智能体。精确、彻底、安全意识。

职责：
- 编码和架构决策
- 代码审查和质量检查
- Bug 调查和修复
- 技术文档

模型：Claude Opus / Codex（用于实现）
频道：Telegram（响应 @dev）

日常任务：
- 检查 CI/CD 流水线健康状况
- 审查开放的 PR
- 标记技术债务项目
```

## 所需技能

- `telegram` 技能用于共享控制界面
- `sessions_spawn` / `sessions_send` 用于多智能体协调
- 共享文件系统或笔记工具用于团队记忆
- 不同模型提供商的单独 API 密钥（如果使用混合模型）
- VPS 或始终在线的机器来运行智能体

## 如何设置

### 1. 共享内存结构

```text
team/
├── GOALS.md           # 当前 OKR 和优先级（所有智能体读取）
├── DECISIONS.md       # 关键决策日志（仅追加）
├── PROJECT_STATUS.md  # 当前项目状态（由所有智能体更新）
├── agents/
│   ├── milo/          # Milo 的私有上下文和笔记
│   ├── josh/          # Josh 的私有上下文
│   ├── marketing/     # 营销智能体的研究
│   └── dev/           # 开发智能体的技术笔记
```

### 2. Telegram 路由

配置一个所有智能体都监听的单个 Telegram 群组，但每个智能体仅在被标记时响应：

```text
## AGENTS.md — Telegram 路由

Telegram 群组："Team"

路由：
- @milo → 战略智能体（生成/恢复 milo 会话）
- @josh → 业务智能体（生成/恢复 josh 会话）
- @marketing → 营销智能体（生成/恢复 marketing 会话）
- @dev → 开发智能体（生成/恢复 dev 会话）
- @all → 广播到所有智能体
- 无标记 → Milo（团队负责人）默认处理

每个智能体：
1. 读取共享的 GOALS.md 和 PROJECT_STATUS.md 以获取上下文
2. 读取自己的私有笔记
3. 处理消息
4. 在 Telegram 中响应
5. 如果响应涉及决策或状态更改，则更新共享文件
```

### 3. 计划任务

```text
## HEARTBEAT.md — 团队计划

每天：
- 上午 8:00：Milo 发布晨会（汇总隔夜智能体活动）
- 上午 9:00：Josh 拉取关键指标
- 上午 10:00：营销从热门主题提出内容想法
- 下午 6:00：Milo 发布当天结束总结

持续进行：
- 开发：监控 CI/CD 健康状况，随到随审 PR
- 营销：Reddit/X 关键词监控（每 2 小时）
- Josh：竞争对手定价检查（每周）

每周：
- 周一：Milo 起草每周优先级（所有智能体的输入）
- 周五：Josh 汇编每周指标报告
```

## 核心见解

- **个性比你想象的更重要**：给智能体不同的名称和沟通风格使得"与你的团队交谈"而不是与通用 AI 较劲变得自然
- **共享内存 + 私有上下文**：这种组合至关重要 —— 智能体需要共同基础（目标、决策），但也需要自己的空间来积累领域专业知识
- **正确的模型用于正确的工作**：不要使用昂贵的推理模型进行关键词监控。将模型能力与任务复杂性匹配
- **计划任务是飞轮**：真正的价值出现在智能体主动呈现见解时，而不仅仅是在你询问时
- **从 2 个开始，而不是 4 个**：从一个负责人 + 一个专家开始，然后在你识别瓶颈时添加智能体

## 灵感来源

这种模式由 [Trebuh 在 X 上描述](https://x.com/iamtrebuh/status/2011260468975771862)，他是一位个人创业者，设置了 4 个 OpenClaw 智能体 —— Milo（战略负责人）、Josh（业务）、一个营销智能体和一个开发智能体 —— 全部通过 VPS 上的单个 Telegram 聊天控制。每个智能体都有自己的个性、模型和计划任务，同时共享项目记忆。他将其描述为"一个真正的 24/7 可用的小团队"。

这种模式也在 [OpenClaw Showcase](https://openclaw.ai/showcase) 上得到了确认，其中 `@jdrhyne` 报告运行"15+ 个智能体，3 台机器，1 个 Discord 服务器 —— IT 构建了大部分内容，只是通过聊天"，而 `@nateliason` 描述了一个多模型流水线（原型 → 总结 → 优化 → 实现 → 重复），在每个阶段使用不同的模型。另一位用户 `@danpeguine` 在同一个 WhatsApp 群组中运行两个不同的 OpenClaw 实例协作。

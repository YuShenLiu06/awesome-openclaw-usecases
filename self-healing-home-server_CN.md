# 自愈家庭服务器与基础设施管理

运行家庭服务器意味着为自己的基础设施全天候 24/7。服务在凌晨 3 点出现故障、证书静默过期、磁盘填满、Pod 崩溃循环 —— 所有都发生在你睡觉或离开时。

这个使用案例将 OpenClaw 转变为一个带有 SSH 访问、自动化 cron 作业以及在你知道有问题之前检测、诊断和修复问题的持久基础设施智能体。

## 痛点

家庭实验室运营商和自我托管者面临持续维护负担：
- 健康检查、日志监控和警报需要手动设置和关注
- 当出现问题时，你必须通过 SSH 登录、诊断和修复 —— 经常从手机
- 基础设施即代码需要定期更新
- 关于你的设置的知识存在于你的脑海中，而不是在可搜索的文档中
- 常规任务（电子邮件分类、部署检查、安全审计）每周占用数小时

## 功能说明

- **自动化健康监控**：基于 cron 的对服务、部署和系统资源的检查
- **自愈**：通过健康检查检测问题并自主应用修复（重启 Pod、扩展资源、修复配置）
- **基础设施管理**：编写和应用 Terraform、Ansible 和 Kubernetes 清单
- **早间简报**：系统健康、日历、天气和任务板状态的每日摘要
- **电子邮件分类**：扫描收件箱、标记可操作项目、归档噪音
- **知识提取**：将笔记和对话导出处理为结构化的、可搜索的知识库
- **博客发布流水线**：草稿 → 生成横幅 → 发布到 CMS → 部署到托管 —— 完全自动化
- **安全审计**：定期扫描硬编码密钥、特权容器和过度许可的访问

## 所需技能

- `ssh` 访问家庭网络机器
- 用于 Kubernetes 集群管理的 `kubectl`
- 用于基础设施即代码的 `terraform` 和 `ansible`
- 用于密钥管理的 `1password` CLI
- 用于电子邮件访问的 `gog` CLI
- 日历 API 访问
- 用于知识库的 Obsidian 保险库或笔记目录
- 用于自我诊断的 `openclaw doctor`

## 如何设置

### 1. 核心智能体配置

在 AGENTS.md 中为你的智能体命名并定义其访问范围：

```text
## 基础设施智能体

你是 Reef，一个基础设施管理智能体。

访问权限：
- 通过 SSH 访问家庭网络上的所有机器（192.168.1.0/24）
- 用于 K3s 集群的 kubectl
- 用于凭据的 1Password 保险库（仅用于读取的 AI 保险库）
- 通过 gog CLI 的 Gmail
- 你的日历（+ 伴侣的）
- ~/Documents/Obsidian/ 处的 Obsidian 保险库

规则：
- 绝不硬编码密钥 —— 始终使用 1Password CLI 或环境变量
- 绝不直接推送到主 —— 始终创建一个 PR
- 作为自我健康检查的一部分运行 `openclaw doctor`
- 将所有基础设施更改记录到 ~/logs/infra-changes.md
```

### 2. 自动化 Cron 作业系统

这个设置的力量在于计划作业系统。在 HEARTBEAT.md 中进行配置：

```text
## Cron 计划

每 15 分钟：
- 检查任务板以获取进行中的任务 → 继续工作

每小时：
- 监控健康检查（Gatus、ArgoCD、服务端点）
- 分类 Gmail（标记可操作项目，归档噪音）
- 检查未回复的警报或通知

每 6 小时：
- 知识库数据录入（处理新的 Obsidian 笔记）
- 自我健康检查（openclaw doctor、磁盘使用、内存、日志）

每日：
- 凌晨 4:00：夜间头脑风暴（探索笔记之间的连接）

每天上午 8:00：早间简报（天气、日历、系统统计、任务板）

上午 10:00：速度评估（流程改进）

每周：
- 知识库 QA 审查
- 基础设施安全审计
```

### 3. 安全设置（关键）

这是不可协商的。在将 SSH 访问授予你的智能体之前：

```text
## 安全检查清单

推送前钩子：
 - 在所有存储库上安装 TruffleHog 或类似的密钥扫描器
 - 阻止任何包含硬编码 API 密钥、令牌或密码的提交

本地优先的 Git 工作流程：
 - 使用 Gitea（自托管）在推送到公共 GitHub 之前用于私有代码
 - CI 扫描流水线在任何公共推送之前运行
 - 主分支合并之前需要人工审查

深度防御：
 - 用于智能体的专用 1Password 保险库（有限范围）
 - 用于敏感服务的网络分段
 - 每天自动安全审计，检查：
   * 特权容器
   * 代码或配置中的硬编码密钥
   * 过度许可的文件/网络访问
   * 已部署镜像中的已知漏洞

智能体约束：
 - 分支保护：主分支需要 PR，智能体无法覆盖
 - 写入不需要的只读访问
 - 所有更改都通过 git 记录并可审计
```

### 4. 早间简报模板

```text
## 每日简报格式

在上午 8:00 生成并交付：

### 天气
- [你的位置]的当前条件和预报

### 日历
- 你今天的所有事件
- 伴侣今天的所有事件
- 标记的冲突或重叠

### 系统健康
- 所有机器上的 CPU / RAM / 存储
- 服务：上/下状态
- 最近的部署
- 过去 24 小时内的任何警报

### 任务板
- 昨天完成的卡片
- 进行中的卡片
- 需要注意的阻塞项目

### 亮点
- 来自夜间头脑风暴的可操作项目
- 需要操作的电子邮件
- 本周的即将截止日期
```

## 核心见解

- **"我现在无法相信自己有一个自愈服务器了"**：智能体可以运行 SSH、Terraform、Ansible 和 kubectl 命令来修复基础设施问题，甚至在你知道有问题之前。
- **AI 会硬编码密钥**：这是 #1 安全风险。如果你不强制保护措施，智能体会高兴地在代码中内联放置 API 密钥。
- **本地优先的 Git 是必不可少的**：绝不要让智能体直接推送到公共存储库。使用自托管的 Gitea 实例作为暂存区域，并进行 CI 扫描。
- **Cron 作业是真正的产品**：计划自动化（健康检查、电子邮件分类、简报）比临时命令提供的每日价值更多。
- **知识提取随时间复合**：处理笔记、对话导出和电子邮件到结构化知识库的价值随时间增长 —— 一位用户单独从他们的 ChatGPT 历史中提取了 49,079 个原子事实。

## 灵感来源

这个使用案例基于 Nathan 的详细写下来 ["到目前为止用 OpenClaw 完成的一切（至今为止）"](https://madebynathan.com/2026/02/03/everything-ive-done-with-openclaw-so-far/)，其中他描述了他的 OpenClaw 智能体 "Reef"，它在一台家庭服务器上运行，通过 SSH 访问所有机器，一个 Kubernetes 集群、1Password 集成，以及一个带有 5,000+ 个笔记的 Obsidian 保险库。Reef 运行 15 个活跃 cron 作业、24 个自定义脚本，并且自主构建和部署了应用程序，包括一个任务管理 UI。Nathan 的第一天 API 密钥暴露后的一个吸取教训："AI 助理会愉快地硬编码密钥。它们有时没有人类一样的直觉。"他的深度防御安全设置对于任何尝试此模式的人来说都是必不可少的阅读。

在 [OpenClaw Showcase](https://openclaw.ai/showcase) 上也引用，`@georgedagg_` 描述了一种类似的模式：部署监控、日志审查、配置修复和 PR 提交 —— 所有都在遛狗时完成。
